You are working inside the DreamTherapy website repo.


Your job is to design and build a minimal, premium, production-ready public website for DreamTherapy that will be deployed on Vercel.


This website is not the iOS app itself. It is the public-facing marketing, trust, support, and compliance website for the DreamTherapy app.


IMPORTANT CONTEXT
- This repo is separate from the iOS app repo on purpose.
- The website must stay simple, minimalistic, and brand-aligned.
- Use the local brand guideline documents in this repo/project folder as the visual and tonal source of truth.
- Use the local image assets in this repo/project folder selectively and tastefully.
- Do not force imagery if the available assets are weak for the homepage.
- The site should feel like DreamTherapy / TEHSO, not like a generic template.


APP CONTEXT TO USE
DreamTherapy is a private dream journaling app with AI-powered reflection.
Its main product pillars/features are:
1. DreamChat
2. DreamBook
3. DreamCircle


These should be treated as the core feature architecture of the app and reflected clearly, simply, and elegantly on the website.


Interpret them like this unless the local docs provide stronger exact wording:
- DreamChat = conversational dream reflection / AI-guided dream dialogue
- DreamBook = private dream journal / saved entries / personal archive
- DreamCircle = community or shared/social dream layer, if present in the product docs


Do not overstate features beyond what is supported by the app docs.
If the local documents define these differently, follow the docs instead.


WEBSITE PURPOSE
This site has five jobs:
1. Clearly explain what DreamTherapy is
2. Make the product feel trustworthy, premium, and emotionally safe
3. Provide public privacy/compliance pages needed for App Store submission
4. Give users a support destination
5. Create a scalable foundation for future marketing expansion


APPLE / COMPLIANCE CONTEXT
Build the website so it supports App Store readiness.
The site must include:
- a public Privacy Policy page
- a public Terms of Use page
- easy access to those pages from the footer and support flow


Write the privacy and terms content carefully and conservatively:
- do not invent fake data practices
- do not make medical or therapeutic claims
- do not say the app diagnoses, treats, or replaces therapy
- do not promise absolute privacy/security unless explicitly verified
- use plain language where possible while still being structured enough for launch use

PRIVACY POLICY PAGE REQUIREMENTS
Create a Privacy Policy page that is:
- cleanly designed
- easy to read
- narrow readable content width
- clearly sectioned
- plain-language but launch-appropriate


It must include at minimum:


1. Introduction
- who DreamTherapy is
- why the policy exists
- note that dream content can feel personal


2. Information we may collect
Use careful, realistic categories such as:
- account information such as email if users create an account
- dream entries and journal content
- voice input/audio if applicable
- usage, analytics, and diagnostics data
- purchase/subscription information if applicable
- support messages


3. How information may be used
Examples:
- provide app features
- save and organize dream entries
- generate reflections
- improve product quality and reliability
- support reminders/notifications if enabled
- provide customer support


4. AI processing
Explain clearly and conservatively that if users submit dream content for AI-generated reflections, that content may be processed to provide those reflections, depending on the app’s setup.


5. Sharing/service providers
Use careful language to explain that information may be processed by service providers used to operate the app, such as hosting, analytics, subscriptions, or AI infrastructure, when applicable.


6. Data retention
Use standard careful language around retaining data as needed to operate the service, comply with obligations, resolve disputes, or enforce agreements.


7. User choices
Include:
- managing account details
- deleting entries
- turning off notifications
- contacting support
- requesting deletion where applicable


8. Security
Use non-absolute language such as reasonable safeguards.


9. Children’s privacy
State the service is not intended for children if that aligns with the app.


10. Changes to this policy


11. Contact information
Use support email placeholder.
